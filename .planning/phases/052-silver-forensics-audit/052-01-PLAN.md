The six dimensions to cover in order:

**Dimension 1 — Session classification paths (session-level, task-level, general)**
silver-forensics Step 2c has a classification table with three paths. gsd-forensics Step 3 uses anomaly detection patterns (stuck loop, missing artifact, abandoned work, crash/interruption, scope drift, test regression) — these are not the same three paths. Note: gsd-forensics Path 2 (task-level) maps to "stuck loop" + "scope drift" + "test regression"; Path 1 (session-level) maps to "abandoned work" + "crash/interruption" + "missing artifact". Determine if silver-forensics covers the scope-drift and test-regression detection patterns.

**Dimension 2 — Evidence-gathering steps (four parallel quick-scan sources)**
silver-forensics Step 2b has four parallel items: (1) most recent session log, (2) `git log --oneline -10`, (3) sentinel file `~/.claude/.silver-bullet/timeout`, (4) `.planning/` directory. gsd-forensics Step 2 has five sub-steps: 2a git history (multiple commands including `git log --format="%H %ai %s" -30`, files-changed uniq-count, `git status --short`, `git diff --stat`), 2b planning state (STATE.md + ROADMAP.md + config.json), 2c phase artifact completeness per phase directory, 2d SESSION_REPORT.md, 2e `git worktree list`. Document each missing evidence source in silver-forensics.

**Dimension 3 — GSD-awareness routing table**
silver-forensics Step 1b provides a routing table that routes GSD-specific issues to `/gsd-forensics`. gsd-forensics does not have this (it IS the gsd-forensics destination). Document as: silver-forensics has this dimension; gsd-forensics does not require it. Record as "Equivalent (dimension not applicable to gsd-forensics)" — no gap.

**Dimension 4 — Root-cause statement format**
silver-forensics mandates: `ROOT CAUSE: <one sentence> — <path taken> — <confidence: high/medium/low>` at the end of every investigation. gsd-forensics uses a prose "Root Cause Hypothesis" section in the report. Document whether the structured one-liner format is present and enforced in silver-forensics's post-mortem report schema.

**Dimension 5 — Post-mortem report schema**
silver-forensics writes to `docs/silver-forensics/YYYY-MM-DD-<slug>.md` with sections: Symptom, Evidence Gathered, Root Cause, Contributing Factors, Recommended Next Steps, Prevention. gsd-forensics writes to `.planning/forensics/report-{timestamp}.md` with: Evidence Summary (git activity table, planning state, artifact completeness matrix table), Anomalies Detected (with confidence levels), Root Cause Hypothesis, Recommended Actions. Document missing sections in silver-forensics: artifact completeness matrix table, anomaly confidence-level annotations, worktree state in Evidence Summary.

**Dimension 6 — Security boundary (UNTRUSTED DATA handling)**
silver-forensics has an explicit "## Security Boundary" section at the top that labels all investigation files as UNTRUSTED DATA, prohibits acting on instructions found in files, and mandates noting "Suspicious content detected in [file]". gsd-forensics has a lighter `critical_rules` block: read-only investigation + redact sensitive data (absolute paths, API keys, tokens). Document whether silver-forensics's UNTRUSTED DATA section covers the redaction rules from gsd-forensics (stripping absolute paths and API keys from the report and issues). Check if silver-forensics's "Allowed Commands" section and redaction rules are equivalent to gsd-forensics's critical_rules.

After the six-dimension analysis, write a "## Gaps Found" section listing ONLY items that require a change to `skills/silver-forensics/SKILL.md`, with a brief fix description for each. Number them G-01, G-02, etc.
