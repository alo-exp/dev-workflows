**Frontmatter** (YAML frontmatter delimiters):

    name: review-cross-artifact
    description: "Cross-artifact consistency reviewer -- validates alignment across SPEC.md, REQUIREMENTS.md, ROADMAP.md, and DESIGN.md. Detects unmapped ACs, orphaned requirements, missing design coverage, and phantom phase requirements."

**Key differences from single-artifact reviewers:**
- Input contract accepts MULTIPLE artifact paths via `source_inputs` array (not a single `artifact_path`)
- Use `artifact_path` for the PRIMARY artifact (SPEC.md -- the source of truth) and `source_inputs` for the others
- Actually, to align with the interface: `artifact_path` = first artifact in the list, `source_inputs` = remaining artifacts. But document clearly that this reviewer expects ALL artifacts in the combined list.
- Better approach: use `artifact_path` as a sentinel value (e.g., ".planning/SPEC.md") and pass all artifact paths via `source_inputs`. Document this convention clearly.

**Input section:** Table with:
| Field | Type | Required | Description |
| artifacts | string[] | YES | List of artifact paths to check consistency across. Accepts: SPEC.md, REQUIREMENTS.md, ROADMAP.md, DESIGN.md |
| check_mode | "full" / "structural" | YES | Defaults to "full" |

Explain: The reviewer auto-detects which artifacts are provided by filename pattern matching (same approach as artifact-reviewer's mapping table). At minimum SPEC.md and REQUIREMENTS.md must be present. ROADMAP.md is expected but optional. DESIGN.md is fully optional (ARVW-09d conditional check).

**Quality Criteria -- 3 independent check groups:**

### QC-1: SPEC to REQUIREMENTS Alignment (ARVW-09b) [content]
- Parse all AC IDs from SPEC.md `## Acceptance Criteria` section (pattern: `AC-XX` or similar identifiers)
- Parse all requirement IDs from REQUIREMENTS.md (pattern: `**REQ-XX**:` or `- [x] **XXX-NNx**:`)
- For EACH AC in SPEC: verify at least one requirement references it (by ID or by content mapping). If not, emit ISSUE finding `XART-F01` "AC {id} in SPEC.md has no corresponding requirement in REQUIREMENTS.md"
- For EACH requirement in REQUIREMENTS.md: verify it traces back to a SPEC AC or is explicitly marked as derived. If orphaned, emit ISSUE finding `XART-F02` "Requirement {id} has no source AC in SPEC.md"
- Suggestion for F01: "Add a requirement in REQUIREMENTS.md that implements AC {id}, or update the AC ID reference in an existing requirement"
- Suggestion for F02: "Map requirement {id} to a SPEC.md AC, or add an AC that this requirement implements"

### QC-2: REQUIREMENTS to ROADMAP Alignment (ARVW-09c) [content]
- Parse all requirement IDs from REQUIREMENTS.md
- Parse all phase `Requirements:` lines from ROADMAP.md (pattern: `**Requirements:** [ID1, ID2]` or `**Requirements**: ID1, ID2`)
- For EACH requirement: verify it appears in at least one phase's Requirements line. If not, emit ISSUE finding `XART-F10` "Requirement {id} is not mapped to any ROADMAP phase"
- For EACH requirement ID referenced in a ROADMAP phase: verify it exists in REQUIREMENTS.md. If not, emit ISSUE finding `XART-F11` "Phase {N} references requirement {id} which does not exist in REQUIREMENTS.md"
- Suggestion for F10: "Add requirement {id} to the appropriate phase's Requirements line in ROADMAP.md"
- Suggestion for F11: "Add requirement {id} to REQUIREMENTS.md, or remove the reference from Phase {N} in ROADMAP.md"

### QC-3: SPEC to DESIGN Alignment (ARVW-09d) [content] -- CONDITIONAL
- **Only evaluate when DESIGN.md is provided in the artifacts list.** If DESIGN.md is not provided, skip this entire check and emit INFO finding `XART-I01` "DESIGN.md not provided -- SPEC-to-DESIGN consistency check skipped"
- Parse user stories from SPEC.md `## User Stories` section
- Parse component/module definitions from DESIGN.md (section headers, component names)
- For EACH user story: verify at least one design component addresses it. If not, emit ISSUE finding `XART-F20` "User story '{story}' has no design coverage in DESIGN.md"
- For EACH design component: verify it relates to at least one user story or AC. If orphaned, emit ISSUE finding `XART-F21` "Design component '{component}' has no corresponding user story or AC in SPEC.md"
- Suggestion for F20: "Add a design component or section in DESIGN.md that addresses this user story"
- Suggestion for F21: "Map this design component to a SPEC.md user story, or remove if it is not needed"

**Structural vs Content tagging:**
- All three QC checks are `content`-tagged (they validate cross-reference consistency, not structure)
- In `structural` check_mode, ALL checks are skipped and the reviewer returns PASS with INFO finding `XART-I00` "Cross-artifact consistency checks are content-level only -- structural mode returns automatic PASS"

**Output Contract section:** Finding IDs use prefix `XART-F` for issues, `XART-I` for info. Same schema as reviewer-interface.md.

**Loading Rules section:** Same pattern as review-spec -- must load reviewer-interface.md and review-loop.md.

**Reviewer Prohibitions section:** Same 5 prohibitions as review-spec.
