    - hooks/session-start (lines 88-112 — the section that calls rm -f trivial, which is the problematic deletion)

  <action>
In hooks/hooks.json, the SessionStart array currently has three entries in this order:
  1. Inline command: `umask 0077 && mkdir -p ~/.claude/.silver-bullet && { [ -L ~/.claude/.silver-bullet/trivial ] && rm -f -- ~/.claude/.silver-bullet/trivial; touch -- ~/.claude/.silver-bullet/trivial; }` (creates trivial)
  2. `session-start` script (which calls `rm -f -- "$trivial_file"` at line 112, deleting trivial)
  3. `spec-session-record.sh`

This means hook 1 creates trivial, then hook 2 immediately deletes it. The trivial file never survives.

Fix: Reorder the SessionStart entries so `session-start` runs FIRST (position 1), then the inline trivial-touch command runs SECOND (position 2), then `spec-session-record.sh` runs THIRD (position 3).

After the fix the execution order is:
  1. `session-start` script runs — does branch-scoped state reset, then calls `rm -f -- "$trivial_file"` (cleaning up any leftover trivial from the PREVIOUS session — this is correct behavior, it's the per-session cleanup)
  2. Inline command: `touch ~/.claude/.silver-bullet/trivial` — creates fresh trivial for THIS session
  3. `spec-session-record.sh`

The inline command in position 2 does NOT need to change (it already creates trivial correctly). Only the ordering of the two JSON objects in the SessionStart array needs to change.

Swap the first two entries in the `hooks.SessionStart` array: the object containing `session-start` becomes index 0, the object containing the inline `touch` command becomes index 1.
  </action>

Line 70 (inside the "branch file absent or empty" branch):

Line 80 (inside the "branch changed" branch):

The stored_branch comparison at line 75 will then work correctly.

Make both changes. No other changes to session-start in this task.

<task type="auto">
  <name>Task 3: Fix BUG-03 — narrow tamper guard in dev-cycle-check.sh to state file only</name>
  <files>hooks/dev-cycle-check.sh</files>
  <read_first>
    - hooks/dev-cycle-check.sh (lines 137-152 — the tamper guard section for Bash commands)
  </read_first>
  <action>
In hooks/dev-cycle-check.sh, the Bash tamper guard regex at line 143 is:

  CURRENT:  `'(>>|\s>[^>&=]|\btee\b)[^<]*\.claude/[^/]+/(state|branch|trivial|mode)'`

This matches writes targeting state, branch, trivial, AND mode — but only the state file should be protected. Writes to branch and trivial are legitimate operations performed by hooks themselves (session-start writes branch, hooks.json writes trivial).

Fix: Change the regex to match only the `state` file, not `branch`, `trivial`, or `mode`.

  FIX:  `'(>>|\s>[^>&=]|\btee\b)[^<]*\.claude/[^/]+/state\b'`

The `\b` word boundary after `state` prevents matching paths like `state_backup` or `state.old`.

The updated guard protects only `~/.claude/<dir>/state` writes from Bash commands. Writes to branch or trivial are no longer blocked.

Note: the `[^/]+` between `.claude/` and `/state` correctly handles both the default path `.claude/.silver-bullet/state` and custom state directories configured in `.silver-bullet.json`.
  </action>
  <verify>
    <automated>grep -n 'state|branch|trivial|mode' hooks/dev-cycle-check.sh | grep -E '\(state\|branch\|trivial\|mode\)' && echo "FAIL: old pattern still present" || echo "PASS: old pattern removed"</automated>

    <automated>bash -n hooks/dev-cycle-check.sh && echo "PASS: syntax OK" || echo "FAIL: syntax error"</automated>
  </verify>
  <acceptance_criteria>

  </acceptance_criteria>
  <done>dev-cycle-check.sh tamper guard matches only writes to the state file; branch and trivial file writes are no longer blocked by this guard.</done>
