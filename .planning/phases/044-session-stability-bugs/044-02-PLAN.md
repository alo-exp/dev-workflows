BUG-05 causes stop-check to block administrative sessions (backlog reviews, documentation-only sessions) that never made any code edits, even when the git tree is clean.

- hooks/completion-audit.sh — cmd classification extracts first line of command only, not full heredoc expansion

  <name>Task 1: Fix BUG-04 — classify git commands from first line only, not full heredoc body</name>

  <action>
In hooks/completion-audit.sh, the command is extracted at line 56:
  `cmd=$(printf '%s' "$input" | jq -r '.tool_input.command // ""')`

This captures the FULL bash command including heredoc body. When Claude runs a command like:

  git commit -m "$(cat << 'EOF'
  ... some message containing ~/.claude/.silver-bullet/state ...
  EOF
  )"

The `$cmd` variable contains all of that text. The grep at line 65 (`grep -qE '\bgit commit\b'`) correctly fires on this. But the body of the heredoc may also contain text matching later gate checks. More critically, the issue described in BUG-04 is: a Bash command whose body contains a state path string causes a FALSE classification as `is_intermediate=true` because the full heredoc is scanned.

Wait — re-reading the bug: the false-positive is `COMMIT BLOCKED`, which is the Tier 1 intermediate commit message. This means `is_intermediate` was set true by `\bgit commit\b` matching inside a heredoc body. For example, a command like:

  cat << 'EOF'
  To reset: git commit --allow-empty
  EOF

Would cause `is_intermediate=true` because `git commit` appears in the heredoc body.

Fix: Extract only the FIRST LINE of the command for classification. Heredoc bodies are never on the first line — the first line is always the actual command being invoked.

After line 56 (`cmd=...`), add:
  `cmd_first_line=$(printf '%s' "$cmd" | head -1)`

Then change all six classification grep calls (lines 65-77) to use `$cmd_first_line` instead of `$cmd`:

  Line 65:  `if printf '%s' "$cmd_first_line" | grep -qE '\bgit commit\b'; then`
  Line 67:  `elif printf '%s' "$cmd_first_line" | grep -qE '\bgit push\b'; then`
  Line 69:  `elif printf '%s' "$cmd_first_line" | grep -qE '\bgh pr create\b'; then`
  Line 71:  `elif printf '%s' "$cmd_first_line" | grep -qE '\bgh pr merge\b'; then`
  Line 73:  `elif printf '%s' "$cmd_first_line" | grep -iqE '\bdeploy\b'; then`
  Line 75:  `elif printf '%s' "$cmd_first_line" | grep -qE '\bgh release create\b'; then`

The rest of the function (which reads state, checks skills, etc.) continues to use `$cmd` for any other purposes. Only the six classification checks use `$cmd_first_line`.

Note: The `[[ -z "$cmd" ]] && exit 0` guard at line 57 remains on `$cmd` (the full command) — do not change that guard.

<task type="auto">
  <name>Task 2: Fix BUG-05 — add admin session detection to stop-check.sh</name>
  <files>hooks/stop-check.sh</files>
  <read_first>
    - hooks/stop-check.sh (full file — understand all existing gates: jq, ERR, no config, trivial, HOOK-14, HOOK-04, required-skills)
  </read_first>
  <action>
In stop-check.sh, the existing HOOK-14 gate (lines 120-185) already handles the "clean tree + no ahead commits" case and exits 0. However, it only exits 0 if `ahead_count == 0`. Sessions that are AHEAD of origin but have no staged/unstaged changes in the working tree — meaning all local commits were already pushed and the session itself made no new edits — still fall through to enforcement.

The success criteria for BUG-05 is: "purely administrative sessions (no Write/Edit calls, no git diff output) bypass enforcement without requiring trivial bypass."

Add a new gate AFTER the HOOK-14 block and BEFORE the state file read (before the `state_contents` assignment at line ~188). This gate detects admin sessions by checking whether the current session produced any actual diff:

Position: after the closing `fi` of the HOOK-14 block (line ~185) and before `# ── Read state file`

Add the following block:

```bash
# ── Admin session detection (BUG-05) ─────────────────────────────────────────
# Sessions with no staged or unstaged changes to tracked files produced no
# code work in this session. Skip enforcement for read-only / conversational
# sessions even when local commits exist ahead of origin.
# Note: this uses --diff-filter=AM to check only Added/Modified tracked files;
# untracked files (which HOOK-14 already handles) are excluded here.
if git -C "$PWD" rev-parse --is-inside-work-tree >/dev/null 2>&1; then
  if git -C "$PWD" diff --quiet HEAD 2>/dev/null && \
     git -C "$PWD" diff --cached --quiet 2>/dev/null; then
    # No staged or unstaged changes to any tracked file in the working tree.
    # This session is administrative (no code edits). Skip enforcement.
    exit 0
  fi
fi
```

This gate exits 0 when:
- `git diff --quiet HEAD` — no unstaged changes to tracked files
- `git diff --cached --quiet` — no staged changes

It does NOT exit 0 if either condition fails (there are untracked files but no tracked changes — untracked files indicate active work). The gate is intentionally conservative: it only skips when BOTH staged and unstaged tracked-file diffs are empty.

Add this block immediately after the closing `fi` of the HOOK-14 block. Add a comment `# ── Admin session detection (BUG-05)` to make it discoverable.
  </action>

  </acceptance_criteria>
