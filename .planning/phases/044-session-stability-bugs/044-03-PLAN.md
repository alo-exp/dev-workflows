**Addition 1: Extend the Rationalization Prevention table**

| Excuse | Reality |

| "It's tracked in the backlog / GitHub issue #N" | A planned fix is not a fix. The violation exists now. ❌ Fail. Fix it or accept the block. |

Add these three rows to the EXISTING Rationalization Prevention table (after the last existing row "The framework forces this structure").

**Addition 2: Add a "Adversarial Mode: No Future Credit" note to the "During Review" subsection**

In `## Applying This Skill > During Review` section, add a new paragraph after the final bullet point (after "No circular dependencies introduced" / "New modules have well-defined interfaces"):

```
**Adversarial mode rule: No future credit.** When running in adversarial mode (pre-ship), the gate evaluates the current state of the code, not plans for future improvement. A violation is ❌ even if:
- It is listed in the project backlog
- A GitHub issue exists tracking it
- The current milestone explicitly plans to fix it in a later phase
- The tech-debt skill was invoked acknowledging it

The only passing condition is that the actual file structure satisfies the rule right now. If a violation is accepted as known tech-debt, mark it ⚠️ N/A with explicit justification — do NOT mark it ✅ Pass.
```

These two additions make the no-future-credit rule explicit and discoverable at exactly the point where Claude evaluates modularity violations.
